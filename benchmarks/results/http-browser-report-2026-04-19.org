#+title: anvil-http + anvil-browser — Cache-Hit Micro-benchmark
#+date: 2026-04-19
#+author: zawatton

* Summary

Small, targeted measurement of the token-saving cache behaviour in
=anvil-http= (Doc 09) and =anvil-browser= (Doc 07), produced on the
same Debian 13 daemon that ran the v1 orchestrator benchmark.
Scope intentionally narrow: two representative URLs, three fetches
each per tool, miss → hit → hit.

** Headline numbers

- *Within TTL, second and third fetches cost 0 ms wallclock and
  zero outbound bytes* for both =anvil-http= and =anvil-browser=.
  The cache hit is served from in-process state; nothing leaves
  the Emacs daemon.  =from-cache= flag on the response plist lets
  a caller distinguish live from cached replies.
- *=anvil-browser= payloads are roughly 7–9× smaller than raw
  HTTP bodies* for the same URL because =agent-browser= renders
  the page and returns an accessibility-tree snapshot instead of
  the raw HTML.  For rendered-content URLs the ratio goes much
  higher (a typical Reddit thread is 400 KB raw HTML vs ~8 KB
  accessibility text).

| URL                           | http bytes | browser bytes | reduction |
|-------------------------------+-----------:+--------------:+----------:|
| =https://example.com/=        |        528 |            73 |    7.23×  |
| =https://httpbin.org/json=    |        429 |            49 |    8.76×  |

** First-call latency (for context, not the point of the bench)

Observed on the author's 1 Gbps home link, Debian 13, Emacs 30:

| URL                        | http miss | browser miss |
|----------------------------+----------:+-------------:|
| =https://example.com/=     |     38 ms |       175 ms |
| =https://httpbin.org/json= |    845 ms |       767 ms |

Browser is slower on static content (agent-browser renders a full
Chromium) but closer-to-parity on JSON endpoints that still route
through the network.  Latency is *not* the selling point of
=anvil-browser=; the fan-out / cleaning / cache is.

* Methodology

=benchmarks/http-browser-v1.el= loops (URL × tool × iter=3),
clearing both module caches before each URL so iter=1 is a true
miss.  Each row records:

- =wall-ms= — end-to-end timing of a single =anvil-http-get= or
  =anvil-browser--tool-fetch= call.
- =body-bytes= — length of the returned body / snapshot text.
- =from-cache= — directly from the =anvil-http-get= response for
  HTTP; for browser, derived from a =anvil-browser--cache-count=
  delta (no entry added ⇒ served from cache).
- =status= — HTTP status code where available.

No LLM tokens are burned for this bench.  The "token reduction"
claim is derived from =body-bytes= on the reasonable assumption
that an agent that would otherwise quote the fetched content back
to the model pays proportional to the bytes it fetched.  Real-task
measurements with a live LLM are deferred to v1.1.

* Caveats

1. *n = 1 per cell.*  Two URLs × three iters × two tools = twelve
   observations.  Variance is not characterised.  Next pass
   should do n ≥ 3 per (URL, tool, iter) and include a third URL
   with heavier content (Reddit thread or long Wikipedia article).
2. *Cache semantics are TTL-based, not etag-based in this sample.*
   =anvil-http= does honour ETags on a 304 response, but neither
   =example.com= nor =httpbin.org/json= returned a useful ETag
   during the bench window; measurement of the ETag code path is
   deferred.
3. *The browser tool's output shape is *accessibility tree*, not
   "clean prose".*  The ratio table above is payload-size, not
   direct content-quality comparable to a rendered article body.
   Agents built on top of =browser-fetch= typically quote the
   snapshot back to the LLM, so payload size is still the right
   proxy for token cost.

* Reproducing

#+begin_src emacs-lisp
  (require 'anvil-http)    (anvil-http-enable)
  (require 'anvil-browser) (anvil-browser-enable)
  (load-file "benchmarks/http-browser-v1.el")
  (anvil-bench-http-browser-run)
#+end_src

The result CSV is written to
=benchmarks/results/http-browser-YYYYMMDD-HHMMSS.csv=.
